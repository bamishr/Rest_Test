1.0 Overview

This documentaion aims to help users understand how to interact with this web API.

2.0 Assumption

In this coffee shop, the types of the coffee and the additions are designed with a specific price (the price of a order could be calculated by the server):

Coffee Price
lattle $ 4.0
macchiato $ 5.35
cappuccino $ 7.5
mocha $ 8.0
Additions Price
hot $ 0.5
shot $ 1.0
none $ 0.0

3.0 Resources and Attributes

This section introduces the Resources and the attributes for them.

Order

Attribute Description Required Input Default Value
OID Identify Key (str) Yes \
type coffee type (str) Yes \
cost price of the oreder (float) \ (automatically returned value)
payment a link to the payment \ (automatically returned link)
additions User's special flavour (str) \ none
status status flag of the order: open, processing, canceled, ready (str) \ open
Pstatus status flag of the payment information: paid, unpaid (str) \ unpaid

Payment

Attribute Description Required Input Default Value
PID Identify Key (str) \ same as OID
Ptype Payment type: cash or card (str) Yes \
amount coffee type (float) Yes \
cardNo price of the oreder (int) \ 0
expireDate a link to the payment (str) \ none
name User's special flavour (str) \ none

4.0 Operations

This sections shows how to use HTTP request to interact with the API. Also, the sample outputs are given.
 On orders:

4.1.1 Creating Order

  • Route: /orders
  • Method: POST
  • Sample Input:

POST http://127.0.0.1:5000/orders?OID=a&type=lattle

  • Sample Output:

201 CREATED
 {
 "OrderID": "a",
"PaymentLink": "/payments/a",
"TotalCost": 4.0
 }
 or (if the input parameters are illegal)
 404 NOT FOUND
 {
 "Input": false
 }

4.1.2 Getting Order Details

  • Route: /orders/<OID>
  • Method: GET
  • Sample Input:

GET http://127.0.0.1:5000/orders/a

  • Sample Output:

200 OK
 {
 "OID": "a",
"Pstatus": "unpaid",
"additions": "none",
"cost": 4.0,
"payment": "/payments/a",
"status": "open",
"type": "lattle"
 }
 or (if the searched ID is not valid)
 404 NOT FOUND
 {
 "ID": false
 }

5.0 Additional Info

  • Every order has two status flags:
    • status flag : open, processing, canceled, ready
    • Pstatus flag: paid, unpaid
  • Cashier use PUT to update an order while Barista use PATCH to change the status of the order to 'processing'.
  • The payment link is given when the order is created with initial Pstatus: 'unpaid'. Cashier only updates(PUT) the payment with the given URI(payment/<PID>) to create it. After that, the Pstatus changes to 'paid'.
  • Barista use GET /payment/<PID> to check the payment information. After checking, the status of the order is changed to 'ready', which means this drink can be released.
  • An order cannot be updated if its status is 'processing' (i.e open orders can be updated). Also, it cannot be updated if the Pstatus is "paid".
